# Local Presentation Coach

Интерактивное демо для **HolyJS 2026**, которое показывает, что **ML-модели можно запускать прямо в браузере пользователя** — без серверов, API-ключей и отправки видео/аудио в облако.

Демо анализирует выступление в реальном времени: взгляд, позу, жесты, мимику и качество звука. После остановки сессии формирует итоговый отчёт с метриками и рекомендациями. Все вычисления выполняются **локально на устройстве**.

**Исходный код:** [github.com/Alice8080/demo-holyjs](https://github.com/Alice8080/demo-holyjs)

---

## Применение

| Что показываем | Почему это важно |
|---|---|
| **Три vision-модели работают одновременно** в live-режиме | Доказывает, что браузер — полноценная ML-платформа, а не только UI-слой |
| **Данные не покидают устройство** | Камера и микрофон обрабатываются в памяти браузера; нет upload, нет стриминга на backend |
| **Нулевая инфраструктура** | Не нужны GPU-серверы, очереди, биллинг за inference, DevOps вокруг ML API |
| **Мгновенная обратная связь** | HUD обновляется каждый кадр — latency определяется только локальным inference, а не сетью |
| **Практичный сценарий** | Не абстрактная классификация картинок, а коучинг выступления — понятный и наглядный use case |

Ответ на вопрос «зачем запускать ML на клиенте»: **приватность, скорость, автономность и отсутствие операционных затрат** — при сохранении качества, достаточного для интерактивного приложения.

## Функционал

### Live-анализ (в реальном времени)

При нажатии «Запустить live-анализ» приложение:

1. Запрашивает доступ к **камере** (1280×720, фронтальная) и **микрофону**
2. Загружает ML-модели и WASM-рантайм (см. раздел «Модели и загрузка»)
3. Запускает цикл обработки кадров через `requestAnimationFrame`

**Vision (ML, MediaPipe):**

| Сигнал | Модель | Что измеряется |
|---|---|---|
| Взгляд | Face Landmarker | Eye contact: положение радужки и поворот головы относительно камеры |
| Поза | Pose Landmarker Lite | Стабильность осанки: наклон плеч, смещение головы |
| Жесты | Gesture Recognizer | Распознавание жестов рук (👍, ✌, 🖐 и др.) с анимацией-реакцией |
| Активность лица | Face Landmarker | Энергия мимики: открытость рта + движение лица между кадрами |

На видео рисуются оверлеи: рамка лица (фиолетовая) и линия плеч (зелёная).

**Audio (сигнальный анализ, Web Audio API):**

| Сигнал | Метод | Что измеряется |
|---|---|---|
| Шум | RMS по Float32-буферу | Слишком тихо / норма / слишком шумно |
| Качество | Детекция клиппинга | Процент кадров с peak > 0.97, рекомендации по gain |

> Аудио-часть использует классический DSP (не нейросеть). Это осознанный выбор: для уровня громкости и клиппинга ML не нужен, а демо фокусируется на **vision-моделях** как главном доказательстве локального ML.

### Итоговый отчёт (после остановки)

При остановке live-анализа:

1. Агрегируются метрики сессии (`sessionStats`) — проценты времени по каждому сигналу
2. Формируется визуальный отчёт с progress-bar метриками
3. Генерируются **эвристические рекомендации** (`buildSessionRecommendations`) — правила на основе порогов, **без LLM и без облачных API**

Примеры рекомендаций: «взгляд уходил мимо камеры 35% времени», «мимика вялая», «обнаружен клиппинг».

## ML-модели: что, откуда, как запускается

### Используемые модели

Все три модели — из экосистемы **Google MediaPipe Tasks**, формат `.task` (TFLite + метаданные), квантизация **float16**:

| Модель | Назначение | URL загрузки |
|---|---|---|
| **Face Landmarker** | 478 лендмарков лица, blendshapes, оценка взгляда и мимики | `storage.googleapis.com/mediapipe-models/face_landmarker/face_landmarker/float16/1/face_landmarker.task` |
| **Pose Landmarker Lite** | Скелет тела (lite-версия для скорости) | `storage.googleapis.com/mediapipe-models/pose_landmarker/pose_landmarker_lite/float16/1/pose_landmarker_lite.task` |
| **Gesture Recognizer** | Классификация жестов рук (до 2 рук) | `storage.googleapis.com/mediapipe-models/gesture_recognizer/gesture_recognizer/float16/1/gesture_recognizer.task` |

### WASM-рантайм

Инференс-движок загружается из npm-пакета через CDN:

```
https://cdn.jsdelivr.net/npm/@mediapipe/tasks-vision@0.10.35/wasm
```

Инициализация в коде (автовыбор backend):

```js
const resolver = await FilesetResolver.forVisionTasks(WASM_FILES_URL)

// WebGL2 доступен → пробуем GPU для всех трёх моделей, иначе CPU
let models
if (isGpuDelegateAvailable()) {
  try {
    models = await createVisionModels(resolver, 'GPU')
  } catch {
    models = await createVisionModels(resolver, 'CPU')
  }
} else {
  models = await createVisionModels(resolver, 'CPU')
}
```

### GPU vs CPU: как именно выполняется inference

| Аспект | Поведение в этом демо |
|---|---|
| **Вычислительный backend** | **GPU (WebGL2)**, если браузер поддерживает WebGL2 и MediaPipe успешно инициализирует delegate; иначе **WASM (CPU)** |
| **Статус в UI** | «Vision-модели: готовы (GPU)» или «Vision-модели: готовы (WASM (CPU))» |
| **Выбор backend** | Проверка `canvas.getContext('webgl2')` → попытка `delegate: 'GPU'` для всех моделей → при ошибке автоматический fallback на `delegate: 'CPU'` |
| **Потоки** | WASM-рантайм MediaPipe использует SIMD и может задействовать несколько потоков через Web Workers (внутри библиотеки) |
| **Режим** | `runningMode: 'VIDEO'` — оптимизирован для последовательных кадров с webcam |
| **Частота** | Один проход трёх моделей на каждый новый кадр видео (`detectForVideo` / `recognizeForVideo`) |

### Что происходит при старте (пошагово)

1. `FilesetResolver.forVisionTasks()` — скачивает и инициализирует WASM-модули (~несколько МБ)
2. Параллельно создаются три экземпляра моделей — каждый скачивает свой `.task`-файл
3. Модели кэшируются браузером (HTTP cache) — повторный запуск значительно быстрее
4. `getUserMedia` → видеопоток на `<video>`
5. `requestAnimationFrame` → на каждом кадре три inference-вызова → обновление HUD и статистики

### Приватность и сеть

| Данные | Куда идут |
|---|---|
| Видеопоток с камеры | Только в RAM браузера, в canvas/video element |
| Аудиопоток | Только в AudioContext, никуда не записывается |
| Метрики сессии | `sessionStats` в памяти, сбрасываются при новом старте |
| Внешние запросы | Только при **первой** загрузке: WASM + веса моделей. Никаких analytics, telemetry, облачного inference |

## Быстрый старт

```bash
npm install
npm run dev
```

Откройте URL из терминала (обычно `http://localhost:5173`), нажмите **«Запустить live-анализ»**, разрешите камеру и микрофон.

```bash
npm run build    # production-сборка
npm run preview  # предпросмотр сборки
```

### Требования

- Современный браузер с поддержкой WebAssembly SIMD, `getUserMedia`, Web Audio API
- Камера и микрофон (для полного опыта)
- Интернет при **первом** запуске (загрузка моделей и WASM)

## Сценарий для живой демонстрации на HolyJS

1. **Старт** — показать, что модели загружаются прямо в браузере (статус «Vision-модели: загрузка... → готовы (GPU)» или «готовы (WASM (CPU))»)
2. **Live** — отойти от камеры → HUD предупредит о взгляде; перекосить плечи → сигнал по позе; показать жест 👍 → emoji-реакция на видео
3. **Аудио** — говорить тихо/громко → HUD по шуму и клиппингу реагирует мгновенно
4. **Стоп** — итоговый отчёт с процентами и рекомендациями формируется **локально за миллисекунды**, без запроса к серверу
5. **Акцент для аудитории:** «Ни один кадр не ушёл с вашего ноутбука. Три нейросети работали параллельно на GPU или CPU прямо в браузере. Backend не нужен.»

## Ограничения

- Модели и WASM загружаются с CDN при первом визите — для полного offline-first можно положить ассеты в `public/`
- Pose Landmarker **Lite** — компромисс скорость/точность; для продакшена можно взять full-модель
- Рекомендации — эвристики, не LLM; для «умного коуча» можно добавить локальный LLM (WebLLM, Chrome Built-in AI), но это другой уровень сложности
